Advantages and Features

Advantages

  • GEDKeeper is a freeware, open-source product without advertisements.
  • It is a cross-platform project; one can build a release for Linux or Microsoft Windows™ operating systems.
  • GEDKeeper does not limit data size.
  • Main functions are always improving; the kernel is continuously developed and tested. We always seek new ways to improve program reliability and performance.
  • Wholesome conservatism of the user interface. The author does not change the GUI at each release—we do not follow new trends blindly.
  • GEDKeeper can be easily extended: it supports scripts and plug-ins. Any user familiar with programming in VB.NET or C# can create a plug-in.
  • GEDKeeper supports several localizations. Any user can create a localization for a native language.
  • Author emphasizes maximum improvement of navigation and search over information in very big data structures of genealogical information (filtering, search, and navigation).
  • GEDKeeper does not utilize genealogical social networks, databases, or cloud services. You are never forced to save data to such storage mediums.
  • GEDKeeper guarantees data safety and integrity in case of possible system errors.
  • You have the freedom in choosing how to process tree and pedigree images. Tree and pedigree images may be exported into a file, then optionally processed, or printed, by third party software. Note: GEDKeeper does provide some printing features.

Disadvantages compared with other open-source, non-profit, and commercial software

  • GEDKeeper does not have a beautiful and stylish interface. It is being built with functionality and performance in mind.
  • GEDKeeper has minimal help content. There are comparativly few users providing material for updated help content.
  • A dedicated website does not exist for an Internet presence.
  • Import of third-party GEDCOM files is being implemented slowly. There is good news: GEDKeeper handles such foreign GEDCOM data very carefully and never loses unknown data structures. While some data may not be shown, they are always saved as-is (see the "Features" section below).
  • There have been claims that GEDKeeper is inflexible. Once it is recognized that the ideology of GEDKeeper is to be a laconic tool for gathering, processing ,and searching information while performing long and broad genealogical research, such claims become unwarranted. The author, for example, has more than 70 research branches in his database used to investigate all possible relatives. Most third-party solutions for tree visualization provide little for effective processing--although they may provide beautiful diagrams. The author and other users work with all the diagrams implemented in GEDKeeper—satisfying their requirements in an efficient and clear way of work. Since this is open source software, the author encourages others to create plug-ins to extend GEDKeeper functionality.

Features

  • GEDKeeper supports nearly all of the GEDCOM specification v5.5.1.
  • GEDKeeper uses extensions to the GEDCOM specification to store and process social groups, researches, tasks, and correspondence. Much genealogical software have such extensions but unfortunately they are all non-standard. There were repeated attempts undertaken in the past to develop a new GEDCOM specification to include these extensions, but all attempts thus far have failed.
  • A non-standard European extension for storing notes and references to geographical locations with coordinates is provided.
  • GEDKeeper saves data using extensions to the GEDCOM standard—yet it retains full compliance.
  • GEDKeeper primarily targets Russian users. Therefore some functions, such as one which handles names, lack some flexability that may not meet the requirements of non-Russian users.
  • Currently GEDKeeper does not store data in a relational database. Experiments performed by the author revealed that a relational database having less than 500,000 person records did not demonstrate benefits in either performance or usability. If a relational database becomes a necessity then only freeware products will be chosen—such as SQLite, MySQL, PostgreSQL, Firebird, or perhaps Berkeley DB.
  • The GEDCOM specification defines two equivalent ways to store internal references to multimedia content, notes, and sources: local storage in the form of an event object and as a full-fledged record—paired to other types of records like persons and families. GEDKeeper uses the latter and converts, as needed, any GEDCOM file which comes from an external source. This instantly makes all notes, multimedia content, and sources available for viewing and navigation. Thus the user can see what records reference those objects.
